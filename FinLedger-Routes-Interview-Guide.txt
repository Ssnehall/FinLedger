  FINLEDGER — Routes Interview Guide
  Backend: Node.js | Express | MongoDB | JWT Auth
================================================================


YOUR PROJECT HAS 3 ROUTE GROUPS
---------------------------------


1. AUTH ROUTES — /api/auth
---------------------------
"I have three auth routes — register, login, and logout.
All of them are POST requests because they involve sending
sensitive data in the request body, not in the URL."

  POST   /api/auth/register
         Creates a new user, hashes password, sends welcome email.

  POST   /api/auth/login
         Validates credentials, issues a JWT stored in a cookie.

  POST   /api/auth/logout
         Blacklists the current JWT token to invalidate the session.


2. ACCOUNT ROUTES — /api/accounts
-----------------------------------
"All account routes are protected — every request must pass
through my auth middleware, which verifies the JWT token and
also checks the blacklist. If the token is invalid or blacklisted,
the request is rejected before even reaching the controller."

  POST   /api/accounts/
         Creates a new bank account for the logged-in user.

  GET    /api/accounts/
         Fetches all accounts belonging to the logged-in user.

  GET    /api/accounts/balance/:accountId
         Returns live balance (computed via ledger aggregation).

  POST   /api/accounts/add-funds
         Deposits money into an account (creates a CREDIT ledger entry).

  PATCH  /api/accounts/:accountId/close
         Closes an account — only allowed if balance is zero.

"I used PATCH for closing an account because I am only changing
one field — the status — not replacing the whole resource.
That is the correct REST convention."


3. TRANSACTION ROUTES — /api/transactions
-------------------------------------------
"I have two transaction routes. The first is for normal
user-to-user transfers. The second is a special system route
for depositing initial funds — it uses a different middleware
that only allows a designated system user to call it, preventing
regular users from minting money out of thin air."

  POST   /api/transactions/
         Transfers money between two accounts — creates one
         Transaction + two Ledger entries (DEBIT + CREDIT),
         all wrapped inside a MongoDB session for atomicity.

  POST   /api/transactions/system/initial-funds
         System-only route to deposit initial funds.
         Guarded by authSystemUserMiddleware — regular users
         cannot access this endpoint.


================================================================
  HOW TO TIE IT ALL TOGETHER — SAY THIS AS A SUMMARY
================================================================

"The routes are split into three routers — auth, accounts, and
transactions — and mounted in app.js under /api/auth, /api/accounts,
and /api/transactions. Every sensitive route goes through the auth
middleware first. The structure follows REST conventions — GET for
reading, POST for creating, PATCH for partial updates."


================================================================
  PRO TIP — LIKELY FOLLOW-UP QUESTION
================================================================

Q: "What does your auth middleware do?"

A: "It reads the JWT from the cookie, verifies it with the secret
   key, then checks the token against the blacklist collection.
   If either check fails, it sends a 401 and stops the request.
   If both pass, it attaches the user object to req.user and
   calls next()."
