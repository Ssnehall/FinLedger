  FINLEDGER — Controllers Interview Guide
  Backend: Node.js | Express | MongoDB | JWT Auth

YOUR PROJECT HAS 3 CONTROLLERS
---------------------------------


================================================================
1. AUTH CONTROLLER — auth.controller.js
================================================================

"The auth controller handles three things — register, login,
and logout."

------------------------------------------------------------
userRegisterController  (POST /api/auth/register)
------------------------------------------------------------
"First I check if a user with that email already exists — if
yes, I return a 422 error. If not, I create the user. Mongoose
automatically hashes the password via the pre-save hook. Then
I sign a JWT token with the user's ID, set it in a cookie,
send the response, and AFTER that I send a welcome email via
Nodemailer. I send the email after the response so the user
is not waiting for the email to send."

------------------------------------------------------------
userLoginController  (POST /api/auth/login)
------------------------------------------------------------
"I find the user by email. I use .select('+password') because
the password field has select: false by default — so I have to
explicitly ask for it. Then I call comparePassword() which uses
bcrypt to compare. If valid, I sign a new JWT and set it in a
cookie. If either check fails, I return the same generic error
— 'Email or password is INVALID' — I do not tell the user
which one is wrong, for security reasons."

------------------------------------------------------------
userLogoutController  (POST /api/auth/logout)
------------------------------------------------------------
"I extract the token from the cookie or the Authorization
header. I save it to the BlackList collection in MongoDB. I
also clear the cookie on the client. Now even if someone has
a copy of this token, it will be rejected by the auth
middleware."


================================================================
2. ACCOUNT CONTROLLER — account.controller.js
================================================================

"The account controller has 5 functions — create, get all,
get balance, add funds, and close account."

------------------------------------------------------------
createAccountController  (POST /api/accounts/)
------------------------------------------------------------
"I get the logged-in user from req.user (attached by auth
middleware). I create the account with the user's ID linked
to it. Default type is SAVINGS and default currency is INR."

------------------------------------------------------------
getUserAccountsController  (GET /api/accounts/)
------------------------------------------------------------
"Simple — I find all accounts where the user field matches
req.user._id. This ensures users can only see their own
accounts."

------------------------------------------------------------
getAccountBalanceController  (GET /api/accounts/balance/:id)
------------------------------------------------------------
"I find the account by both accountId AND req.user._id
together — so even if someone guesses another user's account
ID, they cannot fetch its balance. Then I call
account.getBalance() which runs the MongoDB aggregation
on the ledger collection."

------------------------------------------------------------
addFundsController  (POST /api/accounts/add-funds)
------------------------------------------------------------
"This deposits money. It creates a Transaction record with
only a toAccount (no fromAccount since it's a deposit). Then
it creates a single CREDIT ledger entry for that account.
The idempotency key is auto-generated here."

------------------------------------------------------------
closeAccountController  (PATCH /api/accounts/:id/close)
------------------------------------------------------------
"Before closing, I check two things — first, is it already
closed? Second, does it have a remaining balance? If balance
is greater than zero, I reject the request and tell the user
to transfer funds first. Only if both checks pass do I set
status to CLOSED and save."


================================================================
3. TRANSACTION CONTROLLER — transaction.controller.js
================================================================

"This is the most critical controller. It follows a strict
10-step flow to ensure data integrity."

------------------------------------------------------------
createTransaction  (POST /api/transactions/)
------------------------------------------------------------

STEP 1 — Validate request
  Check all 4 required fields: fromAccount, toAccount,
  amount, and idempotencyKey.

STEP 2 — Check idempotency
  If this key already exists in DB, return the existing
  result — no duplicate processing, no duplicate charges.

STEP 3 — Check account status
  Both accounts must be ACTIVE. FROZEN or CLOSED accounts
  are rejected.

STEP 4 — Check sender balance
  Compute sender's live balance from ledger aggregation.
  Reject if balance is less than the requested amount.

STEP 5 — Start MongoDB session
  All writes below are wrapped in a session for atomicity.

STEP 6 — Create Transaction (PENDING)
  Record the transfer intent inside the session.

STEP 7 — Create DEBIT ledger entry
  Deduct from sender's account inside the session.

STEP 8 — Create CREDIT ledger entry
  Add to receiver's account inside the session.

STEP 9 — Mark COMPLETED + commit session
  Update transaction status to COMPLETED and commit.
  If anything fails before this point, everything
  rolls back automatically.

STEP 10 — Send email notification
  Notify the sender after the session is committed.

"The entire money movement — transaction record, debit entry,
credit entry, and status update — happens inside one MongoDB
session. If any step fails, the session aborts and nothing is
written. That is how I guarantee atomicity."

------------------------------------------------------------
createInitialFundsTransaction  (POST /api/transactions/system/initial-funds)
------------------------------------------------------------
"This is a system-only route. It is used to deposit initial
funds from a designated system user account into any user's
account. It also uses a MongoDB session — creates a DEBIT on
the system account and a CREDIT on the target account,
commits, and returns."


================================================================
  SUMMARY — SAY THIS AT THE END
================================================================

"The controllers follow a clean separation — auth handles
identity, accounts handle money management, and the transaction
controller handles the actual movement of money. The most
complex logic lives in createTransaction which combines
idempotency, balance checks, double-entry ledger writes, and
MongoDB sessions — all in one atomic operation."


================================================================
  PRO TIP — LIKELY FOLLOW-UP QUESTION
================================================================

Q: "What if the server crashes after DEBIT but before CREDIT?"

A: "Since both writes happen inside the same MongoDB session,
   if the session is never committed — which it won't be if
   the server crashes — MongoDB automatically aborts the
   session and rolls back both writes. Nothing is partially
   applied."
